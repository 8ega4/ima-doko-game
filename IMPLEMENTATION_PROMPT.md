# 実装プロンプト — 「いま、どこ？ 消えたボール」MVP

あなたは、スマートフォン向けブラウザゲームを最後まで実装・検証するシニアWebエンジニアです。説明だけで終わらず、このワークスペースに動作するプロジェクトを作成してください。

## ゴール

TikTokやXでプレイ結果を共有したくなる、1プレイ約15〜18秒の1タップゲーム「いま、どこ？ — 消えたボール」のMVPを完成させる。

動いているボールを短時間見せ、その後ボールを非表示にする。内部では同じ速度と角度で動き続ける。「いま、どこ？」と表示した瞬間に物理状態を固定し、プレイヤーが現在位置だと思う場所を1回タップする。正解位置とタップ位置の距離で採点する。

## 最重要体験

- 初見3秒以内にルールが理解できる。
- 操作は各ラウンド1タップだけ。
- プレイヤーの反応速度ではなく、消えたボールの位置予測を測る。
- 大外しもニアミスも一目で分かり、共有したくなる。
- 結果を見た直後に、ロードや確認なしで再挑戦できる。
- 同じシードの軌道を友人に送って、公平にスコア勝負できる。

## 技術構成

- Vite
- React
- TypeScript
- CSS。UIライブラリは入れない。
- Canvas 2D APIでゲームと結果カードを描画する。
- Vitestで物理、シード再現、採点をテストする。
- クライアント完結。API、データベース、ログインは使わない。
- GitHub Pagesへ静的公開できる構成にする。

既存プロジェクトがある場合は構成を確認して流用し、不要な再初期化はしない。新規の場合は最小構成で開始する。

## ゲーム仕様

### 画面

1. タイトル画面
   - タイトル「いま、どこ？」
   - サブタイトル「消えたボールの位置を当てろ」
   - ルールを示す3秒以内の短いデモ、または実線から点線へ変わる小さな図
   - 「プレイ」ボタン
   - 自己ベスト。未プレイ時は非表示。

2. プレイ画面
   - 正方形のフィールド
   - ラウンド表示 `1 / 5`
   - 累計スコア
   - ボール
   - 状態表示「見て」「消えた」「いま、どこ？」
   - 回答後の正解位置、タップ位置、両者を結ぶ線、誤差、得点

3. 結果画面
   - 500点満点の総合得点
   - 称号
   - 5ラウンドの得点と誤差
   - 「もう一回」
   - 「同じ軌道で挑戦」共有
   - モバイル共有
   - X共有
   - 結果画像保存

### 1プレイの状態遷移

`title -> roundIntro -> visible -> hidden -> judgeFrozen -> reveal -> nextRound -> result`

- `roundIntro`: 0.65秒
- `visible`: Round 1の1.3秒からRound 5の1.0秒まで段階的に短くする。
- `hidden`: ボールだけ非表示にし、物理演算は継続する。
  - Round 1: 1.2秒、反射1回を目安、速度1.00倍
  - Round 2: 1.3秒、反射1回を目安、速度1.15倍
  - Round 3: 1.45秒、反射2回を目安、速度1.30倍
  - Round 4: 1.6秒、反射2回を目安、速度1.50倍
  - Round 5: 1.8秒、反射3回を目安、速度1.75倍
- `judgeFrozen`: 「いま、どこ？」の表示と同時にボール位置を固定する。ここから正解位置は動かさない。
- プレイヤーは2秒以内にフィールドを1回タップする。
- 未回答は0点。
- `reveal`: 約1秒。正解と回答を表示し、次ラウンドへ進む。

重要：回答待ちの間にボールを動かさない。反応速度ゲームにしない。

### 物理

- ボールは等速直線運動。
- 壁では入射角と反射角が等しい完全反射。
- 摩擦、重力、ランダムな減速は入れない。
- `requestAnimationFrame`のフレーム数ではなく、経過時間から位置を計算する。
- タブが非表示になった場合はゲームを一時停止し、復帰時に不正な時間ジャンプを起こさない。
- Canvas表示は`devicePixelRatio`に対応し、CSS座標と判定座標を正しく変換する。
- シード付き疑似乱数で初期位置、角度、速度を決める。
- 同じシードでは全端末で同じ5ラウンドを再現する。
- ラウンドごとの反射回数が目標から大きく外れる軌道は、決められた最大試行回数内で再生成する。無限ループは禁止。

### 採点

タップ位置と正解位置の距離をフィールド幅に対する割合へ変換する。

```text
distancePercent = distancePx / fieldWidthPx * 100
roundScore = max(0, round(100 - distancePercent * 2.5))
totalScore = 5ラウンドの合計
```

称号の初期値：

- 500: 完全追跡者
- 470〜499: 預言者
- 420〜469: 物理学者
- 350〜419: 追跡者
- 250〜349: 目で追う人
- 150〜249: 軌道迷子
- 0〜149: ボールと絶縁

称号閾値は定数として一元管理し、後から実測調整できるようにする。

### 演出

- 正解位置は丸、回答位置はバツで明確に区別する。
- 両地点を点線で結び、誤差をpxとフィールド比率で表示する。
- ニアミスでは控えめな成功演出。
- 大外しでは短いシェイクと「どこ見てた？」。
- 演出は短くし、操作可能になるまで待たせない。
- 効果音と振動はユーザー操作後のみ使用する。
- ミュート設定をlocalStorageへ保存する。
- `prefers-reduced-motion`ではシェイクや強い動きを軽減する。

## 共有仕様

### 結果カード

Canvasで1200×1500pxの4:5画像を生成する。

必須要素：

- タイトル
- 総合スコアと称号
- 最も面白い1ラウンドのフィールド
- 見えていた軌道を実線
- 消えていた軌道を点線
- 正解位置を丸
- 回答位置をバツ
- 誤差
- 「あなたなら当てられる？」
- ゲームURLまたは短いドメイン表示

最も面白いラウンドは、原則として誤差が最大のラウンドを選ぶ。ただしニアミスが非常に小さい場合は、自己ベストとしてそちらを優先できる純粋関数にする。

### 共有導線

- `navigator.share`と`navigator.canShare`が画像ファイル共有に対応する場合は、画像と投稿文を共有する。
- X Web Intentではローカル生成画像を自動添付できないため、テキストとチャレンジURLだけを渡す。
- 画像保存ボタンを別に用意する。
- Web Share非対応時はX Intentと画像保存を表示する。
- 共有失敗やキャンセルをゲームエラーとして扱わない。

投稿文テンプレート：

```text
消えたボール、誤差{bestErrorPx}px。
{totalScore}/500点、称号「{title}」でした。
同じ軌道で勝負 → {challengeUrl}
#いまどこゲーム
```

### チャレンジURL

- `?seed=...`で同じ5ラウンドを再現する。
- シード値は安全な長さと文字種に制限する。
- 不正な値は無視して新規シードを生成する。
- URLにスコアを入れて正解扱いにしない。

## 保存

localStorageに保存するのは以下のみ。

- 自己ベスト
- ミュート設定
- 必要なら直近の結果

保存データはバージョン付きにし、壊れたJSONがあっても起動できるようにする。

## UI・レスポンシブ要件

- 縦持ちスマートフォンを最優先。
- 基準サイズは390×844px。
- 320×568pxでも横スクロール、重要要素の欠け、ボタン重なりを発生させない。
- 430px幅まで自然に拡大する。
- PCでは中央にスマートフォン相当のゲーム領域を表示する。
- `touch-action`を適切に設定し、ゲーム操作中のスクロールやダブルタップズーム誤作動を防ぐ。
- セーフエリアを考慮する。
- タップ可能領域は44px以上。
- 色だけに依存せず、丸とバツ、実線と点線で意味を伝える。

## 実装順

1. 既存ファイルと作業ツリーを確認する。
2. 最小のVite + React + TypeScript構成を用意する。
3. シードPRNG、時間基準物理、壁反射を純粋関数として実装する。
4. 物理、シード再現、採点のテストを書く。
5. 1ラウンドのコアループを完成させる。
6. 5ラウンドと結果画面を追加する。
7. localStorageとチャレンジURLを追加する。
8. 結果カード生成と共有導線を追加する。
9. 演出、音、振動、アクセシビリティを追加する。
10. lint、型チェック、テスト、production buildを通す。
11. 390×844、320×568、430×932、PCでブラウザ確認する。
12. 実際に5ラウンドを複数回完走し、再挑戦、同一シード再現、画像生成、共有フォールバックまで確認する。

## 必須テスト

- 同じシードで同じ初期条件・軌道になる。
- 異なるシードで軌道が変わる。
- 各壁で反射方向が正しい。
- 大きなdelta timeでもボールが壁を突き抜けない。
- judgeFrozen後に正解位置が動かない。
- CSS座標とCanvas座標の変換が正しい。
- 距離0で100点になる。
- 得点が0未満、100超にならない。
- 称号境界値が正しい。
- 不正なURLシードを安全に処理する。
- 壊れたlocalStorageでも起動する。
- 同一シードURLで5ラウンドが再現される。

## 完了条件

- 初回アクセスから説明なしでプレイ開始できる。
- 1プレイが10〜30秒以内に収まる。
- 5ラウンドを完走し、結果、再挑戦、共有まで動く。
- 共有画像だけで、実線、点線、正解、回答、誤差が理解できる。
- 同じシードで同じ問題を再現できる。
- スマートフォン幅で横スクロールやUI重なりがない。
- コンソールエラーと未処理Promise rejectionがない。
- lint、型チェック、テスト、production buildが成功する。
- READMEに起動、テスト、ビルド、GitHub Pages公開方法を記載する。

## 作業方針

- 実装前に短い計画を示し、その後は確認待ちで止まらず進める。
- 既存のユーザー変更を上書きしない。
- MVPに不要な機能や抽象化を追加しない。
- 問題が起きたら実際の実行結果を確認して修正する。
- 完了時は、変更ファイル、実装内容、検証結果、残るリスクを簡潔に報告する。
